# گزارش جامع بررسی UI/UX/Responsive پروژه هستو (MVP)

## خلاصه پروژه

اپلیکیشن مالی فارسی (RTL) با نام **هستو — هسته پرداخت ایران**. اپ تک‌صفحه‌ای (SPA) با مسیریابی مبتنی بر Zustand. شامل بخش‌های B2C (کاربر) و B2B (کسب‌وکار) با ۳۵ صفحه B2C و ۱۰ صفحه B2B.

**استک:** Next.js 16, React 19, TypeScript, Tailwind CSS 4, shadcn/ui, Zustand, Vazirmatn font, framer-motion, lucide-react, Recharts

---

## 1. 🚨 مشکلات بحرانی ریسپانسیو و چیدمان

### 1.1 کلاس Tailwind نامعتبر `h-13`
- **فایل:** `src/components/hasto/b2c/screens/transfer-receipt.tsx` — خطوط ۳۷۳ و ۳۸۳
- **مشکل:** `h-13` کلاس استاندارد Tailwind نیست. Tailwind فقط `h-12` (48px) و `h-14` (56px) دارد. نتیجه: ارتفاع دکمه اعمال نمی‌شود و به auto برمی‌گردد.
- **راه‌حل:** تغییر به `h-14` یا `py-3.5` برای ارتفاع مورد نظر.

### 1.2 عدم وجود `overflow-x: hidden` سراسری
- **فایل:** `src/app/globals.css`
- **مشکل:** هیچ `overflow-x: hidden` روی `html` یا `body` تعریف نشده. در اپ RTL این می‌تواند باعث ظاهر شدن اسکرولبار افقی غیرمنتظره شود.
- **راه‌حل:** اضافه کردن `overflow-x: hidden` به `html` یا `body` در globals.css.

### 1.3 کوئری CSS سراسری روی همه inputها
- **فایل:** `src/app/globals.css` — خطوط ۳۵۹-۳۶۵
- **مشکل:** سلکتور `input[type="text"], input[type="tel"], input[type="number"]` با `direction: ltr` و `text-align: right` اعمال می‌شود. این سلکتور **همه** inputهای text را شامل می‌شود و ممکن است روی inputهایی که نیاز به RTL دارند (مثلاً فیلدهای متنی فارسی) تأثیر بگذارد.
- **راه‌حل:** محدود کردن به کلاس خاص (مثلاً `input.persian-number`) یا استفاده از `@apply` با کلاس مشخص.

### 1.4 QR Code با عرض ثابت در صفحه دریافت
- **فایل:** `src/components/hasto/b2c/screens/receive.tsx` — خط ۷۷
- **مشکل:** کانتینر QR کد `w-60 h-60` (240px) دارد. در صفحه‌های کوچکتر از 320px ممکن است overflow ایجاد کند. والد `overflow-hidden` ندارد.
- **راه‌حل:** اضافه کردن `overflow-hidden` به کارت والد یا استفاده از `max-w-full`.

### 1.5 دکمه‌های شناور (FAB) با `fixed` positioning
- **فایل:** `src/components/hasto/b2c/screens/transfer.tsx` — خط ۳۱۹
- **فایل:** `src/components/hasto/shared/ui.tsx` — خط ۱۷۴
- **فایل:** `src/components/hasto/b2c/screens/contracts.tsx` — خط ۲۱۷
- **مشکل:** FABها از `fixed bottom-24 left-1/2 -translate-x-1/2` استفاده می‌کنند. در صفحه‌های بسیار کوچک (< 320px) با `max-w-[420px] w-full px-4` ممکن است overflow ایجاد کند.
- **راه‌حل:** اضافه کردن `overflow-hidden` به کانتینر والد یا استفاده از `safe-area-inset`.

### 1.6 دکمه جستجوی شناور در RTL
- **فایل:** `src/components/hasto/shared/global-search.tsx` — خط ۱۸۸
- **مشکل:** دکمه جستجو `right-4` دارد که در RTL صحیح است (سمت راست = ابتدای RTL). اما در حالت دسکتاپ که فریم تلفن نمایش داده می‌شود، `fixed` positioning نسبت به viewport است نه فریم تلفن، بنابراین دکمه ممکن است از فریم تلفن بیرون بزند.
- **راه‌حل:** استفاده از `absolute` به جای `fixed` یا محدود کردن به کانتینر فریم تلفن.

## 2. ⚠️ باگ‌های UI/UX

### 2.1 تشخیص Dark Mode ناسازگار با SSR
- **فایل:** `src/components/hasto/shared/activity-heatmap.tsx` — خط ۳۰
- **مشکل:** `const isDark = typeof document !== "undefined" && document.documentElement.classList.contains("dark")` — در SSR همیشه `false` برمی‌گرداند. باعث flash رنگ نادرست هنگام hydration می‌شود.
- **راه‌حل:** استفاده از `useTheme()` از next-themes یا `useAppStore((s) => s.theme)`.

### 2.2 سیستم تم دوگانه (Dual Theme)
- **فایل‌ها:** `src/app/layout.tsx` (ThemeProvider از next-themes) + `src/components/hasto/theme-sync.tsx` (مدیریت دستی کلاس dark)
- **مشکل:** دو سیستم مدیریت تم همزمان فعال هستند. ThemeProvider next-themes و ThemeSync هر دو کلاس `dark` را مدیریت می‌کنند. ممکن است تداخل ایجاد شود.
- **راه‌حل:** حذف ThemeSync و استفاده منحصراً از next-themes، یا بالعکس.

### 2.3 تأخیر حذف Toast بیش از حد
- **فایل:** `src/hooks/use-toast.ts` — خط ۱۲
- **مشکل:** `TOAST_REMOVE_DELAY = 1000000` (بیش از 16 دقیقه). این مقدار دیباگ است و در کد باقی مانده. Toastها عملاً هرگز از memory پاک نمی‌شوند.
- **راه‌حل:** تغییر به مقدار معقول مثل `5000` (5 ثانیه).

### 2.4 فرمت اعداد انگلیسی در UI فارسی
- **فایل:** `src/components/hasto/shared/spending-limit-widget.tsx` — خطوط ۵۹، ۸۸، ۱۰۰
- **مشکل:** `limit.toLocaleString("en-US")` و `remaining.toLocaleString()` بدون تبدیل به ارقام فارسی نمایش داده می‌شوند. بقیه UI از تابع `fa()` استفاده می‌کند.
- **راه‌حل:** استفاده از `fa(limit.toLocaleString("en-US"))` برای همه مقادیر عددی.

### 2.5 متن ترکیبی فارسی-اسپانیایی
- **فایل:** `src/components/hasto/b2c/screens/services.tsx` — خط ۹۳
- **مشکل:** `toast.info(`${svc.name} — به زودی disponible`)` — کلمه "disponible" اسپانیایی است، نه فارسی.
- **راه‌حل:** تغییر به `به زودی موجود می‌شود`.

### 2.6 کلیدهای API در کد سمت کلاینت
- **فایل:** `src/lib/hasto-data.ts` — خطوط ۷۹۱-۷۹۲
- **مشکل:** `apiKey: "hasto_live_8f2k9x3m7n4q1w6z5a8b2c9d4e7f0g3h"` و `testKey: "hasto_test_1a2b3c4d5e6f7g8h9i0j"` در داده‌های Mock در کد سمت کلاینت قرار دارند.
- **راه‌حل:** حذف یا جایگزین با متغیرهای محیطی (اگرچه در MVP قابل قبول است).

### 2.7 `ignoreBuildErrors: true`
- **فایل:** `next.config.ts` — خط ۸
- **مشکل:** بررسی خطای TypeScript در build غیرفعال شده. خطاهای نوع پنهان می‌مانند.
- **راه‌حل:** حذف برای production.

### 2.8 کامپوننت Check اختصاصی به جای lucide-react
- **فایل:** `src/components/hasto/shared/daily-streak-widget.tsx` — خطوط ۱۳۹-۱۴۴
- **مشکل:** تابع `Check` اختصاصی با SVG تعریف شده در حالی که بقیه پروژه از `lucide-react` استفاده می‌کند.
- **راه‌حل:** حذف تابع اختصاصی و استفاده از `import { Check } from "lucide-react"`.

### 2.9 تداخل کد تکراری QRCodeSvg
- **فایل‌ها:** `dashboard.tsx` (خط ۶۵۲-۶۸۸)، `receive.tsx` (خط ۴۱۴-۴۴۹)، `receive-with-id.tsx` (خط ۴۵۶-۴۹۱)
- **مشکل:** تابع `QRCodeSvg` سه بار کپی شده.
- **راه‌حل:** استخراج به یک کامپوننت مشترک در `shared/`.

### 2.10 عدم سازگاری `use-mobile.ts` با SSR
- **فایل:** `src/hooks/use-mobile.ts` — خط ۶
- **مشکل:** مقدار اولیه `undefined` است که به `false` ریsolve می‌شود. در اولین رندر همیشه دسکتاپ فرض می‌شود و سپس به موبایل تغییر می‌کند (flash).
- **راه‌حل:** استفاده از `useEffect` یا حذف مقدار اولیه.

### 2.1۱ Badge با overflow-hidden بدون ellipsis
- **فایل:** `src/components/ui/badge.tsx` — خط ۸
- **مشکل:** `overflow-hidden` با `whitespace-nowrap` بدون `text-ellipsis` — متن بلند فارسی بدون علامت بریده می‌شود.
- **راه‌حل:** اضافه کردن `text-ellipsis` یا `truncate`.

## 3. 🚧 مشکلات مسیریابی و سفر کاربر

### 3.1 صفحات بدون کامپوننت (Dead-end در Store)
- **فایل:** `src/components/hasto/b2c/b2c-app.tsx` — خط ۷۹
- **مشکل:** `noChromeScreens` شامل `"payment-pin"` و `"payment-receipt"` است، اما این screenها در رکورد `screens` (خط ۴۴-۷۷) تعریف نشده‌اند. نتیجه: به جای Dashboard fallback می‌شوند (خط ۸۳: `screens[b2cScreen] ?? DashboardScreen`).
- **راه‌حل:** اضافه کردن کامپوننت‌های `PaymentPinScreen` و `PaymentReceiptScreen` یا حذف از `noChromeScreens`.

### 3.2 عدم وجود مسیرهای Next.js
- **فایل:** `src/app/`
- **مشکل:** فقط یک مسیر `/` وجود دارد. تمام صفحات داخلی با Zustand مدیریت می‌شوند. یعنی:
  - رفرش صفحه همیشه به صفحه اولیه برمی‌گردد (البته auth mock هست)
  - URL sharing غیرممکن است
  - Deep linking وجود ندارد
  - SEO وجود ندارد
- **راه‌حل:** برای MVP قابل قبول است. در production نیاز به Next.js App Router routing دارد.

### 3.3 لینک خارجی شکسته (احتمالی)
- **فایل:** `src/components/hasto/b2b/screens/tools.tsx` — خط ۲۰۲
- **مشکل:** `href={business.api.docsUrl}` که مقدار آن `https://docs.hasto.to/api` است. این دامنه احتمالاً وجود ندارد.
- **راه‌حل:** برای MVP استفاده از `#` یا حذف لینک.

### 3.4 `useToast` hook تعریف شده ولی استفاده نشده
- **فایل:** `src/hooks/use-toast.ts`
- **مشکل:** هیچ فایلی این hook را import نمی‌کند. همه فراخوانی‌های toast از `sonner` مستقیماً استفاده می‌کنند. دو سیستم Toast همزمان وجود دارد (sonner + useToast).
- **راه‌حل:** حذف `use-toast.ts` و Toaster از layout، یا یکپارچه‌سازی.

### 3.5 `db.ts` و Prisma غیرفعال
- **فایل:** `src/lib/db.ts`
- **مشکل:** Prisma client تنظیم شده ولی هیچ جا استفاده نمی‌شود. اگر `.env` متغیر `DATABASE_URL` نداشته باشد، import کردن آن در runtime خطا ایجاد می‌کند (البته در حال حاضر import نشده).
- **راه‌حل:** حذف فایل در MVP یا نگهداری برای آینده.

## 4. 💡 پیشنهادات UX برای بهبود حسِ یک محصول واقعی

### 4.1 Empty State برای داشبورد
- صفحه داشبورد هیچ state خالی ندارد. اگر داده‌ها خالی باشند (مثلاً بدون تراکنش)، کاربر صفحه خالی می‌بیند.
- **پیشنهاد:** اضافه کردن `EmptyState` برای لیست تراکنش‌ها و اقساط با آیکون و متن راهنما.

### 4.2 State بارگذاری (Loading)
- هیچ skeleton یا loading state در صفحات وجود ندارد. داده‌ها فوری رندر می‌شوند (Mock).
- **پیشنهاد:** اضافه کردن skeleton برای کارت‌های اصلی داشبورد (wallet, transactions, charts) برای حس واقعی‌تر.

### 4.3 رنگ‌های سفارشی `success`, `warning`, `info`
- **فایل:** `src/app/globals.css` — خطوط ۴۰-۴۴
- **وضعیت:** رنگ‌های `success`, `warning`, `info` در `@theme inline` تعریف شده‌اند اما استاندارد Tailwind نیستند. باید اطمینان حاصل شود که در build نهایی کار می‌کنند.
- **پیشنهاد:** تست build و بررسی عملکرد این رنگ‌ها.

### 4.4 کنتراست رنگ متن کوچک
- فونت‌های `text-[8px]`, `text-[9px]` با `text-muted-foreground` ممکن است WCAG AA را رعایت نکنند.
- **پیشنهاد:** افزایش حداقل اندازه فونت به `text-[10px]` یا تیره‌تر کردن رنگ `muted-foreground`.

### 4.5 حالت تاریک Heatmap
- **فایل:** `src/components/hasto/shared/activity-heatmap.tsx` — خط ۳۰
- **پیشنهاد:** استفاده از `useTheme()` به جای دسترسی مستقیم به `document` برای سازگاری با SSR.

### 4.6 غیرفعال کردن زوم در موبایل
- **فایل:** `src/app/layout.tsx` — خط ۲۸
- **مشکل:** `maximumScale: 1` زوم را غیرفعال می‌کند. برای اپ‌های مالی قابل قبول است اما نگرانی دسترسی‌پذیری دارد.
- **پیشنهاد:** برای MVP قابل قبول. در production بررسی WCAG.

### 4.7 RTL در کامپوننت‌های shadcn/ui
- چندین کامپوننت shadcn/ui از ویژگی‌های فیزیکی (`left`, `right`, `border-l`, `ml-*`) استفاده می‌کنند که در RTL کار نمی‌کنند:
  - `dialog.tsx` خط ۷۲: دکمه بستن `right-4` (باید `left-4` باشد)
  - `sheet.tsx` خط ۷۵: دکمه بستن `right-4`
  - `table.tsx` خط ۷۳: `text-left` در TableHead
  - `sidebar.tsx` خط ۶۴۶: `border-l` در زیرمنو
  - `command.tsx` خط ۱۶۶: `ml-auto` در CommandShortcut
  - `navigation-menu.tsx` خط ۷۸: `ml-1` روی chevron
- **پیشنهاد:** اصلاح این کامپوننت‌ها برای RTL یا استفاده از ویژگی‌های منطقی (`ms-*`, `me-*`, `border-s-*`).

### 4.8 `reactStrictMode: false`
- **فایل:** `next.config.ts` — خط ۹
- **وضعیت:** Strict Mode غیرفعال است (احتمالاً برای سازگاری با framer-motion).
- **پیشنهاد:** بررسی سازگاری framer-motion و فعال‌سازی مجدد.

### 4.9 عدم وجود `aria-label` در برخی المان‌ها
- صفحه Login: input تلفن و OTP فاقد `aria-label` هستند.
- Heatmap: سلول‌ها فاقد `aria-label` هستند.
- **پیشنهاد:** اضافه کردن `aria-label` برای بهبود دسترسی‌پذیری.

### 4.1۰ پیشنهاد ساختاری
- **فایل:** `src/components/hasto/b2c/b2c-app.tsx`
- **پیشنهاد:** اضافه کردن یک صفحه 404 سفارسی برای screenهای ناشناخته به جای fallback به Dashboard.
